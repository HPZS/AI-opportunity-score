#  AI Agent 设计说明

## 一、先用一句话讲清楚这个 Agent

这个 AI Agent 不是一个普通聊天机器人，而是一个面向政企商机发现任务的“任务型智能执行系统”。

它的职责不是陪用户对话，而是围绕明确目标，自动完成：

1. 公开信息检索
2. 网页抓取
3. 线索信号提取
4. 机会初筛
5. 高价值线索深查
6. 结构化结果输出
7. 结果归一化与落盘

“我们设计的不是一个泛化聊天 Agent，而是一个面向政企 AI 商机发现的任务型 Agent。它能按照固定任务流程，自动完成检索、研判、补证据和结构化输出。”

## 二、这个 Agent 为什么要这样设计

这个项目的核心问题是：

- 很多机会不会直接写成 AI 项目
- 公网信息量很大，人工逐条看效率很低
- 线索质量不稳定，容易误判
- 模型如果自由发挥，输出会飘

所以我在设计 Agent 时，没有采用“用户一句话，模型自由发挥到底”的方式，而是采用了“任务约束 + 工具调用 + 结构化输出 + 后处理校正”的设计。

简单说，就是把 Agent 设计成 4 层：

1. 任务层
   - 明确当前到底是在做初筛还是深查
2. 工具层
   - 不是纯靠模型脑补，而是让模型调用检索、抓取、提取、分析工具
3. 结果层
   - 强制输出固定 JSON，而不是自由文本
4. 校正层
   - 后续再做规则归一化、打分补齐、时间窗修正和结果落盘

你在答辩里可以这样讲：

“我们把 Agent 设计成一套受约束的任务系统，而不是完全开放式问答系统。这样做的目的，是让输出更稳定、更可解释、更适合工程落地。”

## 三、Agent 的整体结构

当前 Agent 在项目里的结构，主要由下面几部分组成：

### 1. CLI 入口层

入口文件是：

- [cli.ts](e:/code/mine/AI-opportunity-score/ai-opportunity-agent/src/cli.ts)

这一层负责：

- 读取命令行参数
- 读取 `.env`
- 决定使用 OpenAI 兼容接口还是 Anthropic
- 决定任务类型是 `screening` 还是 `investigation`
- 初始化 Agent
- 调用任务调度器
- 保存最终结果

也就是说，后端并不是直接调用某个 HTTP Agent 服务，而是直接启动一个本地 CLI 进程来执行任务。

## 四、后端是怎么调用 Agent 的

后端调用逻辑在：

- [AgentRuntimeService.java](e:/code/mine/AI-opportunity-score/ai-opportunity-backend/src/main/java/com/aiopportunity/backend/service/AgentRuntimeService.java)

当前实现方式是：

1. 后端构造命令
2. 使用 `ProcessBuilder`
3. 直接执行 `node dist/cli.js`
4. 指定任务类型、输入参数和日志文件
5. Agent 运行后把结果写到本地 `data/task-results`
6. 后端再读取和导入这些结果

所以这个项目里的 Agent 不是独立微服务，而是一个“被后端调起的本地任务执行引擎”。

“为了保链路简单稳定，我们没有把 Agent 设计成独立分布式服务，而是让后端直接拉起本地 Agent 进程。这样系统边界更清晰，演示链路也更短。”

## 五、Agent 内部最核心的设计：任务分层

这个 Agent 不是所有任务混在一起做，而是明确分成两类核心任务：

### 1. 初筛任务 `screening`

作用：

- 从公开信息中发现潜在线索
- 判断哪些线索值得进入候选池

输出重点：

- 场景标签
- AI 适配度
- 商机成熟度
- 是否入池
- 初筛理由

对应配置和模板主要在：

- [screening-config.ts](e:/code/mine/AI-opportunity-score/ai-opportunity-agent/src/config/screening-config.ts)

### 2. 深查任务 `investigation`

作用：

- 不再泛化搜索全部信息
- 只针对高价值线索做补证据和深入分析

输出重点：

- 同源连续性
- 横向案例
- 落地验证
- 政策支撑
- 预算支撑
- 深查结论

对应配置和模板主要在：

- [investigation-config.ts](e:/code/mine/AI-opportunity-score/ai-opportunity-agent/src/config/investigation-config.ts)

“我们把 Agent 工作拆成初筛和深查两层。初筛负责大范围发现机会，深查负责对高价值线索做补证据和可落地性判断。这样做的好处是效率高，而且结果更稳。”

## 六、Agent 为什么不是自由发挥，而是任务模板驱动

在这个项目里，我对 Agent 的一个关键设计，是把任务模板写死。

比如初筛任务里，会明确要求模型：

- 目标池里至少要有多少条线索
- 每轮最多搜几次
- 每次最多返回多少结果
- 必须输出固定 JSON
- 必须分 bucket 输出
- 不能把政策信号误当当前机会
- 不能把 PDF 的弱时间信息直接当最终发布时间

深查任务里也会明确要求：

- 只补证据，不重写初筛主分
- 必须输出 `investigatedLeads`
- 必须输出 `rankedRecommendations`
- 深查分和证据分必须来自工具结果

这种设计的本质是：

- 模型负责理解和归纳
- 系统负责约束边界

## 七、Agent 的执行核心：Agent Loop

Agent 核心循环在：

- [agent.ts](e:/code/mine/AI-opportunity-score/ai-opportunity-agent/src/agent.ts)

它的执行逻辑不是一次调用模型就结束，而是循环式执行：

1. 把系统提示词、任务说明、历史上下文发给模型
2. 模型决定是否调用工具
3. 系统执行工具
4. 把工具结果再喂回模型
5. 模型继续判断下一步
6. 直到输出最终结果

这就是典型的 Agent Loop。

你可以把它讲成一句非常好理解的话：

“模型不是一次性生成最终答案，而是边检索、边抓取、边分析、边修正，最后才形成结构化结论。”

## 八、Agent 里有哪些内部系统

为了让 Agent 不只是“能跑”，而是“跑得稳”，我在内部用了几套系统。

### 1. 工具系统

工具定义在：

- [tools.ts](e:/code/mine/AI-opportunity-score/ai-opportunity-agent/src/tools.ts)

当前业务主链路里最关键的工具有 6 个：

1. `search_web`
   - 调用 Tavily 检索公开网页
2. `fetch_page`
   - 抓取网页正文和元信息
3. `extract_signal`
   - 从网页中提取发布时间、预算、来源类型、场景等信号
4. `screen_opportunity`
   - 对线索做初筛
5. `deep_investigate`
   - 对高潜线索做深查补证据
6. `analyze_opportunity`
   - 输出综合分析结果

也就是说，我不是让大模型直接从一堆网页里自由瞎判断，而是先把任务拆成若干标准动作，每个动作由工具完成。

### 2. 任务监督系统

任务监督逻辑在：

- [task-runner.ts](e:/code/mine/AI-opportunity-score/ai-opportunity-agent/src/task-runner.ts)

这个系统很关键。

它不是“模型跑一次就算完”，而是会检查结果是否达标。

比如初筛任务里，系统会校验：

- 有没有输出合法 JSON
- `summary.poolEntryCount` 是否存在
- 候选池数量是否达到目标值

如果没达到，不会直接失败，而是让任务继续补跑。

“我们不是让 Agent 一次输出完就结束，而是引入了任务监督机制。比如初筛没达到目标池数量，系统会自动继续补跑，直到达标或达到上限。”

### 3. 结果归一化系统

归一化逻辑在：

- [result-normalizer.ts](e:/code/mine/AI-opportunity-score/ai-opportunity-agent/src/result-normalizer.ts)

这是我认为非常有工程价值的一层。

它解决的问题是：

- 模型输出可能不稳定
- 时间窗判断可能冲突
- 某些标题或页面可能是占位页、列表页、预算公开页
- 有些结果不应该误入机会池

所以归一化层会做：

1. 时间窗修正
2. bucket 重分配
3. 弱信号降级
4. PDF 弱发布时间降级
5. 占位标题降级
6. 列表页降级
7. 预算类文件降级
8. 综合分补算
9. 初筛结果与深查结果合并

这个地方非常适合答辩强调：

“模型负责理解，但最终结果不是直接裸用模型输出。我们又加了一层规则归一化，专门负责修正时间、来源、弱信号和综合评分，保证结果稳定可控。”

### 4. 结果存储系统

存储逻辑在：

- [storage.ts](e:/code/mine/AI-opportunity-score/ai-opportunity-agent/src/storage.ts)

它会把结果写到多个目录，而不是只写一个最终文件。

主要包括：

1. `data/task-results`
   - 保存完整任务结果包
2. `data/screening-runs`
   - 保存初筛任务运行结果
3. `data/screening-pool`
   - 保存打平后的候选池
4. `data/investigation-runs`
   - 保存深查任务运行结果
5. `data/investigation-reports`
   - 保存适合展示的深查报告

这样设计的好处是：

- 既能追溯原始任务
- 又能直接给后端导入结构化结果
- 还方便展示和排错

### 5. 容错与恢复系统

当前 Agent 里还保留了任务恢复和失败兜底能力，比如：

- 中断恢复
- investigation 失败时的恢复结果
- 增量保存可入库结果

这意味着任务中途失败，并不一定会把已经拿到的有效结果全部丢掉。

说明这个 Agent 不是“跑崩就什么都没有”，而是具备工程容错思路。

### 6. 通用 Agent 能力底座

这个 Agent 底座本身还具备一些通用能力，例如：

- 会话保存
- 权限控制
- Token 和成本统计
- 上下文压缩
- 子 Agent
- 技能系统

这些能力主要来自：

- [agent.ts](e:/code/mine/AI-opportunity-score/ai-opportunity-agent/src/agent.ts)
- [skills.ts](e:/code/mine/AI-opportunity-score/ai-opportunity-agent/src/skills.ts)
- [subagent.ts](e:/code/mine/AI-opportunity-score/ai-opportunity-agent/src/subagent.ts)

“在业务 Agent 之外，我们还给它保留了通用 Agent 底座能力，比如权限控制、上下文管理和成本控制，所以它不是一个一次性脚本，而是一套可扩展的 Agent 框架。”

## 九、我是怎么设计这个 Agent 工作流的

你是怎么设计这个 Agent 的

### 第一步：先把问题拆成两个任务

不是让 Agent 一次性做完所有事情，而是拆成：

- 初筛
- 深查

### 第二步：给每个任务固定输入输出

不是自由发挥，而是：

- 输入任务说明
- 输出固定 JSON

### 第三步：中间全部用工具驱动

不是直接让模型“看网页自己脑补”，而是强制走：

- 检索
- 抓取
- 信号提取
- 初筛
- 深查
- 综合分析

### 第四步：增加任务监督

不是模型输出一次就结束，而是：

- 不达标继续补跑
- 达标才结束

### 第五步：增加结果归一化

不是直接相信模型，而是用规则再做一轮修正，保证：

- 可解释
- 可入库
- 可展示
- 可追溯

你可以把这段总结成一句话：

“我的设计思路是，把 Agent 从一个开放式对话模型，收敛成一个可控的任务执行器。它每一步都知道自己在做什么，输出也能被后端稳定接住。”

## 十、怎么介绍 Agent 设计

你可以直接按下面这段讲：

“我们这个 Agent 不是通用聊天机器人，而是围绕商机发现任务专门设计的任务型 Agent。  
它内部首先分成初筛和深查两类任务。初筛负责大范围发现机会，深查负责对高价值线索补充证据。  
执行过程中，Agent 不是自由生成结果，而是通过检索、抓取、信号提取、初筛和深查等工具逐步完成任务。  
为了避免模型输出不稳定，我们又增加了任务监督和结果归一化两层机制。任务监督会检查结果是否达标，比如候选池数量够不够；结果归一化则负责修正时间窗、弱信号、列表页、预算公开类文件等问题，并统一补齐综合评分。  
所以整个 Agent 实际上是一个‘模型 + 工具 + 规则 + 存储’的完整执行系统，而不是单纯的一次模型调用。”

## 十一、如果评委追问“为什么这样设计”

你可以这样回答：

“因为这个场景不是创意写作，而是工程任务。  
如果完全依赖模型自由输出，结果会不稳定，也很难入库和展示。  
所以我把 Agent 设计成可约束、可监督、可归一化、可追溯的结构。模型负责理解和判断，系统负责控制边界和稳定性，这样更适合真实业务场景”

## 十二、如果评委追问“你这个 Agent 和工作流引擎有什么区别”

你可以回答：

“工作流引擎更偏固定流程编排，而我们这里的 Agent 在执行过程中会基于当前结果决定下一步该查什么、补什么证据、怎么调整搜索方向。  
所以它不是纯写死的流程，也不是纯自由发挥的聊天模型，而是带有自主决策能力的任务执行系统。”

## 十三、如果评委追问“你这个 Agent 最有价值的设计点是什么”

建议你回答这 3 个：

1. 两阶段任务设计
   - 先初筛，后深查，平衡效率和质量
2. 结构化输出设计
   - 不是自由文本，方便入库和展示
3. 归一化与兜底机制
   - 避免模型误判，提升结果稳定性

## 十四、最后的收尾讲法

你最后可以这样收：

“所以我们这个项目的创新点，不只是用了大模型，而是把大模型真正做成了一个可落地的业务 Agent。它能围绕真实商机发现任务，自动完成检索、研判、补证据、结构化输出和结果校正，最终形成销售和售前可以直接使用的判断依据。”
