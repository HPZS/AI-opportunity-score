# 比赛版 MVP 范围与开发排期建议

## 一、文档目标

本文档用于把项目方案收敛成一版真正可交付的比赛版 MVP，明确哪些功能必须做，哪些功能应该延后，避免研发过程中范围失控。

## 二、比赛版核心目标

比赛版不追求完整平台能力，而是要证明一件事：

系统能够从公开信息中发现政企业务场景中的 AI 改造机会，并输出可跟进的研判结论。

因此比赛版的核心不是“页面多”，而是“链路通”。

## 三、必须跑通的一条主链路

比赛版建议只聚焦以下闭环：

1. 管理员配置关键词
2. 用户手动触发采集任务
3. 后端调用 Tavily 检索公开网页
4. 系统完成去重、来源分类和结构化入库
5. Agent 对线索执行机会初筛
6. 用户对高分线索手动触发深查
7. 系统计算综合评分和时效状态
8. 前端展示线索看板和线索详情
9. 系统生成当日高价值线索简报

只要这条链路能稳定演示，比赛版就成立。

## 四、MVP 功能清单

### P0 必做

1. 关键词配置
   - 支持新增、编辑、启用、停用关键词
2. 手动采集
   - 支持手动触发一次公开信息检索
3. 线索入库
   - 保存标题、链接、来源、时间、摘要、正文、关键词
4. 基础去重
   - URL 去重
   - 标题归一化去重
5. 初筛分析
   - 场景标签
   - AI 适配度
   - 商机成熟度
   - 初筛理由
   - 是否进入候选池
6. 单条深查
   - 对高分线索手动触发深查
7. 综合评分
   - 后端按固定权重聚合总分
8. 时效状态
   - 有效
   - 高时效关注
   - 待复评
   - 已过期
9. 看板页
   - 列表展示、筛选、排序、详情查看
10. 今日简报
   - 展示高分线索与建议动作

### P1 应做

1. 来源可信度规则表
2. Agent 任务日志
3. 深查结果展示页
4. 评分理由结构化展示
5. 近 30 天重新评分入口

### P2 可延后

1. 自动定时采集
2. 多角色权限
3. 高级统计图表
4. 企业画像增强
5. 三模型真实并行评分
6. 批量自动深查
7. 相似内容去重

## 五、明确不建议首版实现的内容

以下内容很容易消耗大量时间，但对比赛版核心闭环帮助有限，建议第一版不要做：

1. 复杂权限体系
2. 完整工作流审批
3. 大而全的数据大屏
4. 多模型真实对比实验平台
5. 全自动批量深查调度
6. 复杂知识图谱能力

这些内容可以写进“后续规划”，不必首版硬做。

## 六、推荐开发优先级

建议按“先后端闭环，再前端体验”的顺序推进。

### 第一阶段：打通数据闭环

目标：
- 能从关键词检索到结构化线索并入库

任务：

1. 建表
   - `keyword_config`
   - `lead_info`
   - `lead_score`
   - `lead_deep_analysis`
   - `source_domain_rule`
   - `agent_task`
   - `agent_task_result`
2. 完成关键词管理接口
3. 完成手动采集接口
4. 完成 Tavily 检索接入
5. 完成 URL 去重与标题归一化去重
6. 完成基础线索入库

交付标志：
- 能在数据库看到结构化线索

### 第二阶段：打通 AI 研判闭环

目标：
- 能给线索打上场景标签并生成评分

任务：

1. 实现 Agent 初筛任务
2. 定义初筛 JSON 契约
3. 完成后端评分聚合逻辑
4. 完成来源可信度加权
5. 完成时效衰减规则
6. 完成深查任务接口
7. 实现单条线索深查

交付标志：
- 能对一条线索输出初筛与深查结论

### 第三阶段：打通产品展示闭环

目标：
- 能把结果稳定展示出来并形成演示页面

任务：

1. 首页总览
2. 线索看板
3. 线索详情页
4. 深查结果展示
5. 今日简报页
6. 关键词配置页

交付标志：
- 能从前端完整演示一条高潜机会的发现与分析过程

### 第四阶段：做比赛演示打磨

目标：
- 提升演示观感和稳定性

任务：

1. 调整高分样例数据
2. 优化评分理由展示
3. 优化异常状态提示
4. 补充演示脚本与操作顺序
5. 准备备用静态数据兜底

交付标志：
- 即使现场网络不稳定，也能完成演示

## 七、建议时间排期

如果按紧凑节奏推进，建议采用下面的排期。

### 第 1 周

重点：
- 统一数据模型
- 建表
- 后端基础接口
- Tavily 接入
- 基础采集入库

目标：
- 线索能采进来

### 第 2 周

重点：
- 初筛 Agent
- 评分聚合
- 时效衰减
- 深查接口

目标：
- 线索能分析并给出结论

### 第 3 周

重点：
- 前端看板
- 详情页
- 简报页
- 配置页

目标：
- 业务闭环能演示

### 第 4 周

重点：
- 联调
- 修复问题
- 打磨演示
- 准备备用方案

目标：
- 比赛现场稳定可展示

## 八、演示重点建议

比赛演示时，不要平均讲所有模块，建议重点突出以下 4 个亮点：

1. 不只找显性 AI 项目，还能识别隐性 AI 改造机会
2. 不是简单搜索，而是“初筛 + 深查”两阶段研判
3. 不是只有分数，还能给出证据、理由和建议动作
4. 结果具备时效性和跟进优先级，不是静态信息库

## 九、主要风险与应对

### 风险 1：外部搜索结果不稳定

应对：

1. 提前准备一批可复用的演示关键词
2. 预留本地样例数据
3. 把失败提示和重试机制做出来

### 风险 2：模型输出不稳定

应对：

1. 强制使用固定 JSON 输出
2. 分维度评分，不直接自由给总分
3. 后端做规则聚合

### 风险 3：功能做得太散

应对：

1. 所有需求都围绕主链路取舍
2. 任何新需求都先问是否影响比赛演示闭环
3. 不影响主链路的功能统一降级到 P1 或 P2

## 十、推荐结论

比赛版最合理的策略不是做“大而全平台”，而是做“一个可完整运行的高价值闭环”。

只要能稳定完成下面这件事，项目就已经具备竞争力：

从公开网页中找出一条隐藏的政企 AI 机会，说明为什么值得做，给出证据、评分和建议动作，并在看板里让销售可以直接判断是否跟进。
