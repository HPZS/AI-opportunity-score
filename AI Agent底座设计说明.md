# 政企 AI 机会搜索分析 Agent 底座设计说明

## 一、文档目的

本文档用于说明“政企垂直场景 AI 应用机会发现平台”中的底层 AI Agent 方案。该 Agent 不负责完整产品能力，不负责前端展示，不负责业务系统管理，而是专注承担以下核心任务：

1. 公开信息检索
2. 机会初筛
3. 机会深查
4. 机会分析
5. 机会评分
6. 结果回传服务端

该 Agent 将作为整个平台中的“AI 搜索分析底座”，为后端服务提供结构化线索结果和评分结果。

## 二、底座定位

本 Agent 的定位不是通用 Coding Agent，也不是完整业务系统，而是一个面向政企业务场景的 `AI 搜索分析 Agent`。

它的核心职责是：

- 从公开互联网信息中发现潜在机会
- 判断某条线索是否属于政企某个业务模块的 AI 改造机会
- 对高潜线索进行深度补证据分析
- 输出结构化分析结果和评分结果
- 将结果推送给后端服务入库

因此，该底座本质上是一个“受控工具循环 Agent”，但工具能力和提示词都服务于“机会发现与研判”场景，而不是代码开发场景。

## 三、技术路线选择

当前方案确定为：**直接基于 `claude-code-from-scratch` 最小源码进行改造**。

相关仓库链接：

- `claude-code-from-scratch`：
  <https://github.com/Windy3f3f3f3f/claude-code-from-scratch>

- `how-claude-code-works`（源码解读文档仓库）：
  <https://github.com/Windy3f3f3f3f/how-claude-code-works>

本项目在本地仓库中的底座目录统一命名为：`ai-opportunity-agent`，用于承载后续政企 AI 商机分析 Agent 的定制化改造代码。
当前项目实现层面只保留 TypeScript 版本，Python 版本不纳入本项目后续开发与维护范围。

选择原因如下：

1. 该项目已经具备最小可运行 Agent 的核心能力，包括：
   - Agent 主循环
   - 工具调用机制
   - 会话管理
   - 子 Agent 能力
   - 多轮上下文处理

2. 我们当前对它的使用目标非常明确：
   - 不需要保留 Coding Agent 的代码编辑能力
   - 不需要保留复杂的编程工具链
   - 只需要把它改造成 AI 搜索、分析、评分 Agent

3. 因此，相比完全从零重写，基于最小 Agent 源码改造更快，更适合当前阶段作为底层能力原型。

## 四、总体架构

该底座采用“初筛 / 深查双链路 + 机会生命周期管理 + 结果回传”的结构。

后续正式形态不再把所有动作混在一个任务里，而是明确拆成两个核心动作：

1. `screening`
   - 面向公开信息做广泛召回、抓取、信号抽取和机会初筛
   - 目标是产出结构化候选池，而不是直接给出最终跟进结论

2. `investigation`
   - 只针对初筛后的高潜线索执行深查
   - 目标是补证据、判断优先级、生成最终建议动作

整体流程如下：

1. 接收任务
   - 服务端下发初筛任务或深查任务
   - 任务可带订阅、信号源、时间窗、TopN 等参数

2. 初筛链路
   - 通过搜索工具获取候选信息
   - 抓取正文并抽取结构化信号
   - 判断是否存在潜在 AI 改造机会
   - 形成候选池

3. 去重、归并与状态更新
   - 对重复网页做幂等处理
   - 对同一机会不同阶段做归并
   - 对历史机会新增事件做状态更新或重启判断

4. 深查链路
   - 从候选池中选出待深查对象
   - 围绕高潜机会继续搜索和验证
   - 补充同源连续性、同类案例、落地验证、政策与预算支撑等证据

5. 机会分析与排序
   - 提炼场景标签
   - 判断业务模块
   - 判断 AI 适配方式
   - 结合初筛结果、深查结果和场景热度生成最终建议动作

6. 结果回传
   - 将结构化结果发送给后端服务
   - 或先写入本地 JSON，供比赛版 / 原型期使用

当前原型期在上述链路中额外增加两层工程兜底能力：

1. 模型先输出结构化 JSON
2. 程序侧再对关键字段做一致性纠偏
3. 如果深查任务中途因上游异常中断，程序侧先落盘部分结果检查点
4. 最终保存规范化后的结果，并保留原始文本输出用于回溯

这层规范化当前主要用于解决以下问题：

- 时间窗判断与 `publishTime` 冲突
- 线索被错误放入 `currentOpportunities`
- `summary` 统计与实际分桶不一致
- 深查做到一半因模型空响应或抓取失败导致结果全丢

因此，当前架构中“模型负责生成候选判断，程序负责结果一致性收口与异常恢复”。

## 五、两阶段分析机制

### 1. 机会初筛

机会初筛用于从大量公开信息中快速找出“可能存在 AI 改造机会”的候选线索。
初筛只回答“哪些线索值得进入候选池”，不直接替代深查判断。

初筛关注：

- 是否包含明显业务场景信号
- 是否存在采购、建设、升级、试点、政策推动等信息
- 是否涉及政府、国企或重点企业客户
- 是否属于公司关注的业务模块

初筛产出：

- 是否进入候选机会池
- 初筛理由
- 初筛分
- 基础场景标签
- 入池分层
  - `优先跟进`
  - `观察入池`
  - `信号跟踪`
  - `不入池`
- 信号分类
  - `明确招采`
  - `前置信号`
  - `方向信号`
  - `参考线索`
- 线索分层结果
  - 当前商机
  - 历史案例
  - 政策信号
  - 时间窗外线索

补充说明：

当前原型期已不再把“进入候选池”严格等同于“已有预算或正式采购公告”。只要满足以下条件，线索即可进入候选池持续跟踪：

1. 方向与目标场景匹配
2. 时间窗有效
3. 主体真实可信
4. 存在持续建设、持续运营或明确关注的信号

因此，候选池中既可以有“马上跟进”的机会，也允许保留“方向正确但证据尚弱”的观察型线索。

### 1.1 初筛任务定位

`screening` 的任务边界如下：

1. 负责面向全网或指定信号源召回候选线索
2. 负责生成候选池，不负责给出最终成交概率判断
3. 负责产出后续深查的输入对象

推荐输入：

- 主题
- 关键词订阅 ID
- 信号源预设 ID
- 时间窗
- 召回数量
- 初筛阈值

推荐输出：

- 一次初筛任务结果
- 一个候选池快照
- 不同语义线索的分层列表
- 一组运行统计信息
  - `searchQueryCount`
  - `searchResultCount`
  - `uniqueSearchResultCount`
  - `fetchPageCount`
  - `extractSignalCount`
  - `screenedLeadCount`
  - `acceptedLeadCount`
  - `rejectedLeadCount`

### 2. 机会深查

机会深查只针对高潜线索执行，用于回答“这是不是一个值得认真跟进的真实机会”。
深查不是重新做一遍初筛，而是围绕少量候选对象补齐决策证据。

深查重点包括：

- 同源连续性
  - 同一单位或同一发布源是否连续出现类似信号

- 横向同类案例
  - 其他地区、其他政府、其他企业是否出现类似需求

- 落地验证
  - 是否已有采购、中标、试点、上线、实施案例

- 政策与预算支撑
  - 是否有专项资金、政策推动、财政支持等依据

深查产出：

- 深查证据摘要
- 深查分
- 深查结论
- 是否建议正式跟进
- 证据分类链接
  - `sourceLinksByType.main`
  - `sourceLinksByType.sourceContinuity`
  - `sourceLinksByType.similarCases`
  - `sourceLinksByType.landingCases`
  - `sourceLinksByType.policySupports`
  - `sourceLinksByType.budgetSupports`
- 时间线
- 排序推荐列表
- 深查运行统计

### 2.1 深查任务定位

`investigation` 的任务边界如下：

1. 只处理候选池中的指定线索或 TopN 线索
2. 重点补充“是否值得投入售前资源”的证据
3. 输出单条深查报告和批次深查榜单

推荐输入：

- 来自某次初筛任务的 leadId 列表
- 或“从候选池按规则选择 TopN”
- 深查证据维度配置
- 深查阈值

推荐输出：

- 单条深查报告
- 批次深查结果
- 最终推荐清单
- 批次级运行统计
- 必要时输出“部分完成但可续跑”的检查点结果

### 2.2 深查中断恢复机制

由于 `investigation` 任务依赖上游模型与联网抓取，当前原型期已额外加入“中断检查点 + 自动续跑”机制。

设计原则如下：

1. 如果深查任务执行到一半，因上游空响应、网络抓取失败等问题中断，不直接丢弃已完成结果。
2. 程序侧会优先从已执行的工具结果中恢复出“部分深查结果”。
3. 部分结果按 `partial` 状态落盘，并标记：
   - `taskMeta.taskState = partial`
   - `taskMeta.resumable = true`
   - `taskMeta.resumeKey`
4. 部分结果中需显式区分：
   - `investigatedLeads`
   - `pendingLeads`
   - `resumeInfo`
5. 后续再次以同一任务输入启动深查时，系统会自动识别最近一次未完成检查点，并将其作为续跑上下文注入任务提示，优先补剩余线索，而不是从零开始重做。

### 3. 标准链路建议

建议后续把该底座标准化为如下链路：

1. 执行初筛任务
2. 形成候选池
3. 做去重、归并与状态更新
4. 从候选池中筛出待深查对象
5. 对 TopN 线索逐条深查
6. 输出最终重点机会榜单

如果要“深查 10 个机会”，正确流程应是：

1. 先召回 30 至 100 条候选
2. 初筛后留下 10 至 20 条高潜对象
3. 再选 Top 10 执行深查
4. 由深查结果决定最终排序，而不是只看初筛分

## 六、多模型分工设计

本底座支持“不同能力由不同模型承担”。

建议采用如下分工：

### 1. 初筛模型

用于：

- 候选线索识别
- 基础场景判断
- 初步机会筛选

要求：

- 成本较低
- 响应较快
- 具备较强文本理解能力

### 2. 深查模型

用于：

- 多条检索结果综合理解
- 补充证据整理
- 同类案例归纳
- 落地验证分析

要求：

- 推理能力较强
- 长上下文能力较好

### 3. 分析模型

用于：

- 业务场景抽象
- AI 适配方式判断
- 建议动作生成
- 综合结论归纳

要求：

- 较强业务归纳能力
- 输出稳定、结构化

### 4. 评分模型

评分阶段采用 3 个模型分别独立评分：

- 评分模型 A
- 评分模型 B
- 评分模型 C

每个模型独立输出：

- 采购意向强度
- 客户匹配度
- 预算信号
- 时效性
- AI 适配度
- 综合评分
- 评分理由

最终评分计算方式：

- 取三个模型综合评分的平均值，作为最终总分
- 同时保留三个模型的原始评分，便于解释和回溯

## 七、推荐评分机制

建议评分维度至少包含以下几项：

1. 采购意向强度
   - 判断该线索距离真实项目有多近

2. 客户匹配度
   - 判断该客户和场景是否符合公司当前方向

3. 预算信号
   - 判断是否存在预算、资金、建设投入依据

4. 时效性
   - 判断当前是否处于适合跟进的时间窗口

5. AI 适配度
   - 判断该业务模块是否适合采用 AI 提效或替代

最终分数应支持：

- 单模型原始分
- 三模型平均分
- 是否高潜机会标记

## 八、与服务端的协作方式

该 Agent 不直接承担前端展示，而是作为后端能力模块存在。

推荐交互方式如下：

1. 服务端下发任务
   - 关键词检索任务
   - 单条线索分析任务
   - 高潜线索深查任务

2. Agent 执行分析
   - 完成初筛、深查、分析、评分

3. Agent 回传结果
   - 通过 HTTP API 将结构化结果发送到服务端

建议回传字段包括：

- 线索标题
- 原始链接
- 来源站点
- 来源类型
- 发布时间
- 场景标签
- 初筛分
- 深查分
- 综合分
- 三模型评分明细
- 评分理由
- 建议动作
- 时效状态

### 8.1 Agent 与后端职责边界

为了避免底座能力和业务后端职责重叠，建议明确以下边界。

#### Agent 负责

1. Tavily 检索与网页抓取
2. 线索信号提取
3. 场景识别
4. 机会初筛
5. 机会深查
6. 输出结构化结果与证据摘要

#### 后端负责

1. 关键词与来源规则管理
2. 采集任务调度
3. 去重、归并与幂等控制
4. 来源可信度分级
5. 总分聚合与时效衰减
6. 数据入库与状态流转
7. 对前端提供查询接口

边界原则如下：

1. Agent 不直接写业务数据库。
2. Agent 返回统一 JSON 契约，由后端完成校验和入库。
3. 后端保留最终业务状态解释权，Agent 不承担状态流转职责。

### 8.2 结果稳定性设计

底座输出要面向业务系统使用，因此不能依赖自由文本结果。建议采用以下机制保证结果稳定：

1. 所有分析结果必须按固定 JSON Schema 输出。
2. Agent 只输出分维度分数和证据，不自由生成最终总分。
3. 最终总分由后端按照固定权重聚合。
4. 每个分值都必须附带证据摘要、判断理由和置信度。

推荐分维度输出包括：

- 场景标签
- 场景置信度
- AI 适配度
- 商机成熟度
- 深查补证据得分
- 建议动作
- 模型置信度

#### 8.2.1 当前原型期的稳定性落地方式

在当前实现中，结果稳定性不是只靠提示词约束，而是采用“模型输出 + 程序纠偏”的双保险方式。

当前已落地规则如下：

1. 模型输出的 JSON 不直接作为最终结果保存。
2. `screening` 结果在保存前，必须经过程序侧规范化处理。
3. `screening` 任务除保留主分桶外，还统一补齐运行统计与入池分层字段。
4. `investigation` 任务在正常完成时输出完整报告，在异常中断时输出可恢复检查点。
5. 程序侧会统一校正：
   - `withinTimeWindow`
   - `currentOpportunities / historicalCases / policySignals / outOfWindowLeads` 的分桶归属
   - `summary.currentOpportunityCount`
   - `summary.historicalCaseCount`
   - `summary.policySignalCount`
   - `summary.outOfWindowCount`
   - `summary.poolEntryCount`
   - `summary.recommendedFollowUpCount`
6. 如果 `publishTime` 无法支撑“当前可跟进”的判断，则该线索当前原型期不进入主商机列表，按保守策略降入非主列表处理。
7. 原始模型输出文本仍需保留，便于后续回放、排查和 Prompt 调优。
8. 如果深查任务只完成了一部分线索，程序侧会保留：
   - 已完成的 `investigatedLeads`
   - 待继续的 `pendingLeads`
   - `resumeInfo`
   以便下次续跑。

这意味着当前版本里，以下字段属于“程序校正后的最终落盘结果”，而不是单纯相信模型原文：

- `withinTimeWindow`
- 线索所在 bucket
- `summary.*`
- `poolEntryTier`
- `opportunitySignalClass`
- 深查中断时的 `pendingLeads / resumeInfo`

### 8.3 数据质量前置约束

Agent 在输出结果时，应尽量补全以下数据质量字段，供后端使用：

1. 标准化 URL
2. 标题归一化结果
3. 发布时间原始文本
4. 发布时间归一化结果
5. 发布时间抽取置信度
6. 来源域名
7. 来源类型建议值
8. 时间窗判定结果
9. 当前分层归属

这些字段不一定都由 Agent 最终裁定，但必须作为结构化信号回传，方便后端统一处理。

### 8.4 双配置、双存储建议

为了避免初筛和深查逻辑相互污染，建议底座在配置和存储层也做明确分层。

#### 1. 配置分层

建议拆成两套配置：

1. 初筛配置
   - 关键词订阅
   - 信号源预设
   - 初筛提示词
   - 初筛阈值
   - 召回和时间窗规则
   - 场景识别规则

2. 深查配置
   - 深查提示词
   - 深查证据维度
   - TopN 规则
   - 深查阈值
   - 最终建议动作规则

#### 2. 存储分层

建议也拆成两套数据目录：

1. 初筛数据
   - `task-results`
   - `screening-runs`
   - `screening-pool`

2. 深查数据
   - `task-results`
   - `investigation-runs`
   - `investigation-reports`

比赛版没有后端时，也建议遵守这一分层，先落本地 JSON。

补充说明：

1. `task-results` 用于保存每次任务的完整落盘结果，便于回放和排错。
2. `screening-runs` / `investigation-runs` 用于保存按任务类型拆分后的运行结果。
3. `screening-pool` 用于保存可供后续深查消费的候选池快照。
4. `investigation-runs` 在当前原型期除保存正常深查结果外，也承担“中断检查点”的存储职责。
5. `investigation-reports` 用于保存深查报告正文；若任务中断，则可先保存部分报告结果。
6. 当前原型期以本地 JSON 为主，不直接写业务数据库。

补充建议：

后续可进一步将深查运行状态统一抽象为：

- `completed`
- `partial`
- `failed`

其中 `partial` 明确表示“已有部分可复用结果，允许自动续跑”，避免把可恢复异常与彻底失败混在一起。

### 8.5 去重、重启与同类机会聚类

本项目中的“去重”不能简单理解为“见过就丢掉”，而应拆成以下三层：

#### 1. 原始记录去重

目标是解决同一网页被重复召回的问题。

建议规则：

- URL 标准化去重
- 标题归一化去重
- 同域名同标题去重

#### 2. 同一机会归并

目标是把同一客户、同一项目、不同阶段的记录归并为同一个机会主实体。

典型阶段包括：

- 采购意向
- 招标公告
- 中标公告
- 合同公告

这些记录不能简单删除，而应作为同一机会的不同事件保留。

#### 3. 老机会重启

如果半年前发现过的机会，在本月再次出现且状态显著升级，则不应被旧的去重规则吞掉，而应触发重启。

推荐状态：

- `new`
- `update`
- `reactivate`

推荐重启条件：

- 机会阶段升级
- 首次出现预算、招标、项目编号等强信号
- `shouldEnterPool` 从 `false` 变为 `true`
- `isActionableNow` 从 `false` 变为 `true`

#### 4. 同类机会聚类

不同客户出现同类机会时，不应去重，也不应并成同一个机会。

正确做法是：

- 保留为多个独立机会
- 同时归入一个“同类机会簇”
- 将横向扩散情况作为深查和排序依据

因此，后续建议采用三层对象：

1. `lead`
   - 具体机会主实体
2. `leadEvent`
   - 机会不同阶段和不同时间的事件记录
3. `opportunityCluster`
   - 同类场景簇，用于表达市场热度和扩散趋势

### 10.2 后续工具能力增强方向

在双链路方案下，后续建议补充以下非大模型能力：

### 10.1 比赛版工具集裁剪建议

比赛版不建议一开始就把所有工具全部做全，可按以下顺序实现：

1. `search_web`
2. `fetch_page`
3. `extract_signal`
4. `screen_opportunity`
5. `deep_investigate`
6. `analyze_opportunity`
7. `push_result`

当前原型已额外补齐以下底座能力：

1. PDF 正文抓取与解析
2. 时间窗显式传参与窗口过滤
3. 结果保存前的程序级规范化
4. 初筛 / 深查双配置、双存储
5. 初筛任务运行统计输出
6. 初筛放宽入池策略与分层入池标记
7. 深查异常中断检查点与自动续跑

因此，比赛版虽然不追求完整业务后端，但必须保证“能抓到、能分层、能纠偏、能复盘”。

以下工具可先保留接口，不要求首版真实落地：

- `score_model_b`
- `score_model_c`
- `aggregate_scores`

第一版可由单模型输出分维度结果，再由后端完成综合分聚合。

多模型评分仍是推荐终态，不是当前原型的强制门槛。

## 十一、建议运行模式

该底座推荐支持三种模式：

### 1. 初筛任务模式

由服务端或人工发起 `screening` 任务，Agent 自动完成：

- 搜索
- 抓取
- 初筛
- 形成候选池

### 2. 深查任务模式

由服务端或人工发起 `investigation` 任务，Agent 自动完成：

- 从候选池读取目标线索
- 补证据
- 深查
- 输出重点清单

如果任务执行中发生上游异常，当前原型期应优先：

1. 保存部分深查结果
2. 标记待补线索
3. 下次按相同任务输入继续续跑

### 3. 单条深查模式

针对某条候选线索进行人工触发式深查，适合售前或销售对重点项目做专项分析。

### 4. 兼容模式

为兼容比赛版或当前原型实现，可临时保留“一次任务内先初筛再深查”的组合模式，但长期目标仍应是双链路分离。

补充说明：

当前实现已明确以 `screening` 和 `investigation` 两个独立任务类型为主，组合模式只作为兼容方案，不作为主链路继续扩展。

## 十二、结论

当前方案完全可以基于最小 Agent 源码改造实现，而且适合当前阶段作为“AI 搜索分析底座”。

该底座的价值不在于替代业务系统，而在于为业务系统提供稳定的 AI 机会发现、深查分析和多模型评分能力。其最终定位是：

**一个面向政企 AI 应用机会发现任务的底层 Agent 引擎。**

更准确地说，它最终应是：

**一个支持初筛、深查、去重、重启和同类机会聚类的机会生命周期管理型 Agent 底座。**

在当前原型阶段，可进一步具体表述为：

**一个以 `screening` / `investigation` 双链路为核心，具备搜索、抓取、PDF 解析、结构化信号提取、结果规范化、运行统计输出、深查中断恢复和本地可回放存储能力的 AI 搜索分析 Agent 底座。**
